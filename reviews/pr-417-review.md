## Blocking findings

### B1 - AI-attribution regex false-positives on bare URL/domain matches

`plugins/imbue/hooks/vow_no_ai_attribution.py:31-32`

```python
r"claude\.ai",
r"anthropic\.com",
```

These patterns have no attribution-context anchors. They match any commit
message that mentions the URL in any role: dependency notes, dead-link
removals, integration documentation, or (the case we have evidence for)
commit messages that describe the regex itself. Commit `a2d7ca5d`'s body
contains the literal string `'patterns, plus anthropic.com URL and "by
Anthropic" prose match'` and would have triggered the hook in shadow mode.

Compare with the well-anchored sibling patterns at lines 27-28:

```python
r"Co-authored-by:\s*.*\b(Claude|Anthropic)\b",
r"Generated\s+by\s+(Claude|Anthropic)\b",
```

Suggested direction: scope the URL patterns to attribution contexts (e.g.,
`r"(?:Co-authored-by|Generated by|Reviewed-by).*\b(?:claude\.ai|anthropic\.com)\b"`)
or require they appear in the last `N` lines of the message (trailers).

### B2 - 50-thread atomicity test passes without flock due to GIL serialization

`plugins/imbue/tests/unit/hooks/test_vow_bounded_reads.py:631-672`

The test uses `threading.Barrier` to release 50 threads simultaneously, then
asserts `sorted(results) == list(range(1, 51))`. The PR description claims
this test "reliably fails" without `fcntl.flock`. It does not.

CPython's GIL serializes the JSON parse-modify-write inside `_atomic_increment`
at the bytecode level. The `os.read`/`os.write` calls release the GIL only
briefly, and on tmpfs the entire RMW completes in under 1 ms. Threads almost
never interleave inside the unprotected window. The monkeypatch test at
line 675 explicitly confirms `_HAS_FCNTL=False` increments correctly when
called sequentially - which proves that the unfenced path is not catastrophically
broken, but the *concurrent* test would also pass on that path most of the
time.

This is a test-quality issue, not a correctness issue with the fix itself.
The `fcntl.flock` call is genuinely useful on systems where the GIL doesn't
help (PyPy, free-threaded CPython 3.13t, or any future GIL removal). But the
test as written cannot tell whether flock is doing anything.

Suggested direction: inject a deterministic race by patching the RMW to sleep
between read and write (`time.sleep(0.01)` inside a monkeypatched
`_atomic_increment`), then assert correctness with and without flock.
Alternatively, downgrade the PR claim from "reliably fails" to "documents intent."

### B3 - 30-thread reset-interleave test asserts a trivially-satisfied range

`plugins/imbue/tests/unit/hooks/test_vow_bounded_reads.py:800-808`

```python
final = _read_counter(counter_path)
assert 0 <= final <= n_increments
```

This range is satisfied by any 32-bit integer the file contains. More
importantly, `_read_counter` is fail-safe: on a JSON parse error it returns
`0`. So even a fully corrupted file (`{"count": 0{"count": 1}}` or empty)
satisfies the assertion. Reverting `_atomic_reset` to a non-locked write
would still pass nearly 100% of the time.

Suggested direction: read the raw bytes after the run and assert valid
single JSON document; or count parse failures during the run and assert
zero; or compare against a known-corrupt baseline.

## Related work landed during review

While preparing this review I discovered and fixed an executable-bit
regression on the four new vow hooks (commit `508651eb` on this branch):
they were committed at mode `0644` so `Permission denied` was raised on
every `PreToolUse` since v1.9.1. The hooks were silently inactive. Issue
[#420](https://github.com/athola/claude-night-market/issues/420) was filed
to add `check-shebang-scripts-are-executable` to `.pre-commit-config.yaml`
to prevent recurrence. An audit of all 20 commits since `c8539940` found
**zero** AI-attribution or emoji-commit violations that the inactive hooks
should have blocked (one borderline meta-match in `a2d7ca5d`'s message
which is the same evidence cited in B1).
