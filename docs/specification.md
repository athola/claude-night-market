## Overview

**Purpose**: Add hybrid hard-wrapping (80 chars, sentence-boundary
preference) and consistent formatting conventions to all markdown
documentation generated by claude-night-market plugins.

**Scope**:
- **IN**: All skills that generate/modify markdown prose (scribe,
  sanctum, attune, spec-kit); shared formatting module; rules
- **OUT**: Retroactively reformatting all existing documentation
  (incremental adoption); external tool integration (mdformat,
  prettier); hook-based automated reformatting

**Stakeholders**:
- Plugin developers: need readable diffs when reviewing doc changes
- AI agents: need clear formatting instructions in skills
- Mobile reviewers: need lines short enough to avoid horizontal
  scrolling on GitHub mobile

## Functional Requirements

### FR-001: Shared Markdown Formatting Module

**Description**: Create a leyline shared module that defines the
canonical markdown formatting rules. All doc-generating skills
reference this module rather than duplicating rules.

**Acceptance Criteria**:
- Given the module exists at
  `plugins/leyline/skills/markdown-formatting/SKILL.md`,
  when any doc-generating skill needs formatting guidance,
  then it references `Skill(leyline:markdown-formatting)`
  or includes a cross-reference to the module path
- Given the module content,
  when reviewed by a developer,
  then it contains: wrapping rules, exemption list, structural
  rules, and examples for each rule

**Priority**: High
**Dependencies**: None (foundation for all other FRs)
**Estimated Effort**: M

---

### FR-002: Hybrid Text Wrapping at 80 Characters

**Description**: Prose text blocks in markdown must wrap at 80
characters or fewer per line. Wrapping prefers sentence boundaries
(after `.` `!` `?`), then clause boundaries (after `,` `;` `:`),
then before conjunctions (and, but, or), then at word boundaries.

**Rules**:
1. If a sentence is <= 80 chars, keep it on one line
2. If a sentence is > 80 chars, break at the nearest clause
   boundary before column 80
3. If no clause boundary exists before column 80, break at the
   last word boundary before column 80
4. Never break inside a word
5. Never break inside inline code (backtick spans)
6. Never break inside link text or link URLs

**Acceptance Criteria**:
- Given a paragraph with sentences under 80 chars,
  when formatted,
  then each sentence appears on its own line
- Given a paragraph with a 120-char sentence containing a comma
  at position 65,
  when formatted,
  then the line breaks after the comma
- Given a paragraph with a 100-char sentence containing no
  punctuation,
  when formatted,
  then the line breaks at the last word boundary before column 80
- Given a line containing `[link text](https://very-long-url...)`,
  when the link pushes the line beyond 80 chars,
  then the link is NOT broken mid-URL (line may exceed 80 chars
  or reference-style is used per FR-006)

**Priority**: High
**Dependencies**: FR-001
**Estimated Effort**: M

---

### FR-003: Content Type Exemptions

**Description**: Certain markdown content types must be exempt
from the 80-char wrapping rule because wrapping them would break
rendering or readability.

**Exempt content types**:
1. Tables (pipe-delimited rows)
2. Code blocks (fenced with triple backticks or indented 4 spaces)
3. Headings (lines starting with `#`)
4. YAML/TOML frontmatter (between `---` or `+++` delimiters)
5. HTML blocks
6. Link reference definitions (`[id]: url`)
7. Image references (`![alt](url)`)
8. Blockquotes containing any of the above

**Acceptance Criteria**:
- Given a table row exceeding 80 chars,
  when formatting rules are applied,
  then the table row remains on one line
- Given a code block with lines exceeding 80 chars,
  when formatting rules are applied,
  then code block content is untouched
- Given a heading exceeding 80 chars,
  when formatting rules are applied,
  then the heading remains on one line
- Given a frontmatter block,
  when formatting rules are applied,
  then frontmatter content is untouched

**Priority**: High
**Dependencies**: FR-001, FR-002
**Estimated Effort**: S

---

### FR-004: Blank Lines Around Headings

**Description**: Every heading must be preceded by a blank line
and followed by a blank line. Exception: the first line of a file
may be a heading without a preceding blank line.

**Acceptance Criteria**:
- Given a heading with no blank line before it,
  when formatting rules are applied,
  then a blank line is inserted before the heading
- Given a heading with no blank line after it,
  when formatting rules are applied,
  then a blank line is inserted after the heading
- Given a heading at the start of a file,
  when formatting rules are applied,
  then no blank line is inserted before it

**Priority**: Medium
**Dependencies**: FR-001
**Estimated Effort**: S

---

### FR-005: ATX-Only Headings

**Description**: All headings must use ATX style (`# Heading`).
Setext-style headings (underlined with `===` or `---`) must not
be generated.

**Acceptance Criteria**:
- Given a doc-generation skill producing headings,
  when markdown is generated,
  then all headings use `#` prefix syntax
- Given instructions to write a heading,
  when the skill follows formatting rules,
  then setext style is never used

**Priority**: Medium
**Dependencies**: FR-001
**Estimated Effort**: S

---

### FR-006: Reference-Style Links for Long URLs

**Description**: When an inline link would push a line beyond
80 characters, use reference-style link syntax instead. Place
link definitions at the bottom of the current section or at the
end of the document.

**Acceptance Criteria**:
- Given a prose line with an inline link where the line
  exceeds 80 chars,
  when formatted,
  then the link uses reference-style syntax with the definition
  placed after the current section
- Given a prose line with a short inline link where the line
  is under 80 chars,
  when formatted,
  then the inline link syntax is preserved
- Given multiple references to the same URL,
  when formatted,
  then a single reference definition is used

**Priority**: Medium
**Dependencies**: FR-001, FR-002
**Estimated Effort**: S

---

### FR-007: Blank Line Before Lists

**Description**: A blank line must appear before the first item
of any list (ordered or unordered). This prevents rendering
ambiguity in some parsers and improves source readability.

**Acceptance Criteria**:
- Given text immediately followed by a list item (no blank line),
  when formatting rules are applied,
  then a blank line is inserted before the list
- Given a blank line already present before a list,
  when formatting rules are applied,
  then no additional blank line is added

**Priority**: Medium
**Dependencies**: FR-001
**Estimated Effort**: S

## Non-Functional Requirements

### NFR-001: Compatibility - No Rendered Output Changes

**Requirement**: Applying the formatting rules to existing
markdown must not change the rendered HTML output.

**Measurement**:
- Metric: Diff of rendered HTML before and after reformatting
- Target: Zero differences
- Tool: Manual comparison or `pandoc` rendering

**Priority**: Critical

---

### NFR-002: Performance - No Agent Slowdown

**Requirement**: Following the formatting rules must not
noticeably slow down documentation generation by agents.

**Measurement**:
- Metric: Additional agent turns required for formatting
- Target: Zero additional turns (formatting happens inline)
- Tool: Observation

**Priority**: High

---

### NFR-003: Maintainability - Single Source of Truth

**Requirement**: Formatting rules must be defined in one shared
module. All skills reference this module rather than duplicating
the rules.

**Measurement**:
- Metric: Number of independent rule definitions
- Target: 1 (the shared leyline module)
- Tool: Grep for formatting rule text across skills

**Priority**: High

## Technical Constraints

- **No external tools**: Skills generate markdown directly via
  Write/Edit tools. No dependency on mdformat, prettier, or
  other formatters.
- **Instruction-based**: The formatting is enforced through skill
  instructions that agents follow, not through automated
  post-processing.
- **Python 3.9 compatibility**: If any validation hooks are added,
  they must work on Python 3.9.6.
- **Incremental adoption**: Existing docs do not need to be
  bulk-reformatted. New and modified docs follow the new rules.

## Out of Scope

- Automated reformatting of all existing documentation
- Pre-commit hooks that auto-wrap markdown (future enhancement)
- Integration with external tools (mdformat, prettier, Vale)
- Wrapping rules for non-markdown content (YAML, JSON, Python)
- Changes to code block formatting or indentation rules

**Rationale**: The primary goal is to establish formatting
conventions that agents follow during generation. Automated
enforcement and retroactive reformatting can be added later
as separate initiatives.

## Dependencies

- leyline plugin (shared module host)
- All doc-generating plugins: scribe, sanctum, attune, spec-kit
- Existing directory-style-rules module in sanctum

## Glossary

- **Hybrid wrapping**: Line wrapping that prefers semantic
  boundaries (sentences, clauses) over arbitrary character
  positions
- **ATX heading**: Heading syntax using `#` prefix
  (`# Heading 1`, `## Heading 2`)
- **Setext heading**: Heading syntax using underlines
  (`===` for H1, `---` for H2)
- **Reference-style link**: Link syntax that separates the URL
  from the text: `[text][ref]` with `[ref]: url` defined
  elsewhere
- **Prose text block**: Paragraphs of flowing text, as opposed
  to tables, code blocks, lists, or other structured content
- **Rewrap cascade**: When a single word change causes multiple
  subsequent lines to reflow due to hard wrapping
